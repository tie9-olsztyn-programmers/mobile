# Snackbar

:::note Snackbar
- to krótkotrwały komunikat interfejsu użytkownika:
    - pojawia się na dole ekranu, aby poinformować użytkownika o wyniku wykonanej akcji,
    - nie przerywa pracy programu

- kiedy używać Snackbar:
    - po zapisaniu danych („Zapisano zmiany”),
    - po wykonaniu akcji („Dodano do listy”),
    - przy drobnych błędach („Brak połączenia z siecią”).
:::

## Kluczowe pojęcia
- `SnackbarHostState()`:
    - obiekt stanu,
    - zarządza wyświetlaniem Snackbarów:
        - kolejka komunikatów,
        - widoczność snackbara
    - to nie UI, a logika sterująca UI

- `SnackbarHost`
    - miejsce na ekranie dla Snackbara

- `showSnackbar()`
    - funkcja wyświetlająca Snackbar

- `Scaffold`
    - gotowy szkielet strony
    - każdy element aplikacji na swoim miejscu
    - metaforycznie:
        - fundamenty, ściany nośne, otwory na okna i drzwi

- `rememberCoroutineScope`
    - narzędzie pozwalające bezpiecznie uruchamiać zadania w odpowiedzi na zdarzenia
    - wyświetlanie Snackbara jest asynchroniczne
    - „pomostem” umożliwiającym wykonanie zadania asynchronicznego po kliknięciu jest obiekt `CoroutineScope`
      zwracany przez funkcję `rememberCoroutineScope`

:::tip Korutyny
- „lekki wątek”, lekki mechanizm wykonywania zadań współbieżnych
- pozwalają „poczekać na coś” w kodzie, nie zatrzymując interfejsu użytkownika
- *co-operative routines* – współpracujące procedury
- sposób na pisanie kodu asynchronicznego
- przykład (`launch` start korutyny)
```kotlin
scope.launch {
    delay(2000)
    println("minęły 2 sekundy")
}
```

:::

:::tip `CoroutineScope`
- to "obszar", w którym korutyny mogę bezpiecznie działać
- zasada Kotlina *No coroutine withoit a scope*
- metafora *CoroutineScope* jest jak przełożony, który:
    - pozwala uruchomić zadanie,
    - pilnuje, kiedy je przerwać,
    - odpowiada za jego sprzątnięcie.
    - Korutyna nie może działać „na dziko”.

:::

## Snackbar w użyciu
```kotlin

class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()
        setContent {
            SnackBarDialogAlertDialogTheme {
                val snackbarHostState = remember { SnackbarHostState() }
                val scope = rememberCoroutineScope()


                Scaffold(modifier = Modifier.fillMaxSize()) { innerPadding ->

                    Column(
                        modifier = Modifier
                            .fillMaxSize()
                            .padding(innerPadding)
                            .padding(24.dp)

                    ) {
                       Button(onClick = {
                           scope.launch {
                               snackbarHostState.showSnackbar(
                                   message = "Hejka! Tu SnackBar",
                                   actionLabel = "To jest actionLabel",
                                   duration = SnackbarDuration.Short
                               )
                           }
                       }){
                           Text(text="Pokaż Snackbara")
                       }
                        Spacer(modifier = Modifier.height(10.dp))
                        TextField(
                            value = "coś",
                            onValueChange = {println("coś wpisujesz")},
                            label = { Text("to jest lable do TexField")}
                        )

                        SnackbarHost(hostState = snackbarHostState)
                    }
                }
            }
        }
    }
}
```

## Mała poprawka ze `Scaffold`
Scaffold zarządza układem całego ekranu
- `TopBar`
- `BottomBar`
- `FAB` (*Floating Action Button*, przycisk "zrób to teraz" w prawym dolnym rogu (np. "utwórz notatkę")
- `Content`
- `Snackbar`

**Snackbar jest częścią `layout`u `Scaffold`, a nie zwykłym elementem `Column`** - więc:
- skasuj ` SnackbarHost(hostState = snackbarHostState)`, które jest ostatnim elementem w `Column`
- `Snackbar` trafi do `Scaffold`:
```kotlin
    Scaffold(
         modifier = Modifier.fillMaxSize(),
         snackbarHost = {
            SnackbarHost(hostState = snackbarHostState)
         }
    ) { innerPadding ->   //...
```

# Toast

Wyobraź sobie, że aplikacja to restauracja.

Toast jest jak kelner, który przechodzi obok i rzuca krótkie: „Smacznego!”, nie czekając na Twoją reakcję.

Snackbar jest jak kelner, który stawia sól na stole i mówi: „Gdyby było za mało słone, proszę użyć tej soli” – daje Ci informację i możliwość interakcji (przycisk akcji).

:::info Toast
- w Compose używamy niemal identycznie jak w starym systemie XML, ponieważ wywołujemy go poprzez Context.
- Kluczową różnicą jest sposób pozyskiwania tego kontekstu.

:::




### W aplikacji z **XML**:
```kotlin
// Fragment kodu w starym stylu
button.setOnClickListener {
    Toast.makeText(this, "Wysłano wiadomość!", Toast.LENGTH_SHORT).show()
}

```

### W aplikacji z **Compose**:
```kotlin
@Composable
fun ToastExample(modifier: Modifier = Modifier){
    val context = LocalContext.current // pobieramy kontekst

    Column(modifier=modifier.fillMaxSize(),
        verticalArrangement = Arrangement.Center,
        horizontalAlignment = Alignment.CenterHorizontally
        ){
        Button(onClick = {
            Toast.makeText(
                context,
                "To jest Toast w Compose",
                Toast.LENGTH_LONG
            ).show()
        }){
            Text("Pokaż tosta!!!")
        }
    }
}
```


### Czym jest Kontekst (Context) w systemie Android?
#### Wyjaśnienie metaforyczne
- Wyobraź sobie, że system Android to ogromna, luksusowa galeria handlowa,
a Twoja aplikacja to jeden ze sklepów w tej galerii.
- `context` to twój identyfikator pracowniczy (identyfikator dostępu) połączony z krótkofalówką.
  - Identyfikator: Bez niego nie możesz wejść do magazynu, żeby pobrać zasoby
(np. teksty tłumaczeń z pliku strings.xml czy obrazki z folderu drawable).
  - Krótkofalówka: Bez niej nie możesz skontaktować się z dyrekcją galerii (systemem Android),
aby poprosić o przysługę, np. o wyświetlenie globalnego powiadomienia (nasz wcześniejszy Toast),
uruchomienie innej aplikacji (np. przeglądarki) czy sprawdzenie, czy telefon ma połączenie z internetem.

Krótko mówiąc: `Context` mówi systemowi Android:
- kim jesteś,
- gdzie aktualnie się znajdujesz w aplikacji i
- do czego masz prawo.

#### Wyjaśnienie techniczne
- Z inżynieryjnego punktu widzenia, Context to abstrakcyjna klasa bazowa w systemie Android.
- Działa ona jako "pomost" (interfejs) między Twoim kodem a globalnym środowiskiem systemu operacyjnego.
- Kontekst zapewnia dwie główne funkcjonalności:
  - Dostęp do zasobów (Resources):
  - Pozwala pobierać pliki, ciągi znaków (strings), kolory, style czy motywy przypisane do Twojej aplikacji.
  - Dostęp do usług systemowych (System Services):
  - Pozwala uruchamiać nowe ekrany (Activity),
  - wysyłać intencje (Intent),
  - zarządzać powiadomieniami, czy
  - odczytywać stan baterii.

W starym systemie XML (View System), widoki takie jak `Button` czy `TextView`
"żyły" wewnątrz aktywności (Activity), która sama w sobie dziedziczyła po klasie Context.
Dlatego często mogliśmy po prostu wpisać `this`.

W Jetpack Compose funkcje kompozytowalne (`@Composable`) to zwykłe funkcje Kotlina,
które nie wiedzą nic o cyklu życia systemu Android.
Dlatego, aby uzyskać dostęp do tej "krótkofalówki",
musimy jawnie poprosić o nią system, używając `LocalContext.current`.
